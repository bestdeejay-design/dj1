## 🔄 ЖИЗНЕННЫЙ ЦИКЛ ДОКУМЕНТАЦИИ

### Обновление при изменениях кода

**Минорные изменения (исправление багов):**
- → Обновляем историю в README.md
- → Добавляем запись в changelog
- → Если критичная функция — добавляем метку ⚠️ CRITICAL в код

**Мажорные изменения (новая фича, рефакторинг):**
- → Обновляем CODE_CHEATSHEET.md (новые функции)
- → Обновляем AUDIT.md (архитектурные изменения)
- → Обновляем README.md (история)
- → Добавляем метки 🔒 LOCKED для зависимостей

**Критичные изменения (исправление проблем):**
- → Все вышеперечисленное
- → Выделяем раздел "BREAKING CHANGES"
- → Пишем миграционный гайд
- → Добавляем комментарии 💡 ПОДСКАЗКА в код

## 📝 ПРИНЦИПЫ ВЕДЕНИЯ ДОКУМЕНТАЦИИ

### ✅ ДЕЛАЕМ
1. **Обновляем существующие файлы** вместо создания новых
2. **Добавляем примеры кода** для сложных моментов
3. **Пишем для людей** а не для галочки
4. **Ведём историю** всех изменений
5. **Ссылаемся друг на друга** для навигации
6. **Добавляем метки в код** для критичных мест:
   - ⚠️ CRITICAL — важная функция, трогать осторожно
   - 🔒 LOCKED — есть зависимости, не нарушать порядок
   - 💡 ПОДСКАЗКА — комментарий для разработчика
7. **Проверяем CODE_CHEATSHEET** перед изменениями

### ❌ НЕ ДЕЛАЕМ
1. Не создаём дублирующие файлы
2. Не пишем очевидные вещи
3. Не забываем обновлять после изменений кода
4. Не используем сложные термины без пояснений
5. Не храним устаревшую информацию
